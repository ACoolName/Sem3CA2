Goals

The goal of this CA was to create a web-site which is capable of serving a Business Academy needs for keeping information for people in the Academy. They are also assigned diffrent roles like Student or Teacher. As additional goals courses and class rooms with time blocks should be added. The system must also know about the relations between the diffrent "entities". Besides the web-site as a front-end, it must use REST which calls JPA internally. The database used must be Oracle.

Design description

The project was split in relatively independent parts. From bottom to top we have the a persistence.xml file which contains database specific details. It is read by JPA to persists the entity objects. In a separate package "entities" we have all Entity objects which respond to the tables, provide an ORM interface for the upper layers and are handled by the entity manager from JPA. These objects must be presented as JSON so a facade that makes the convertions and convinience methods is provided in the facades package. A "dto" package is used by the facade to provide only what is "needed" to the upper layers. For example a Person contains a list of roles, but it's not needed by the REST service so an intermediate DTO which does not contain the list is returned. From there the REST server uses the facade to answer requests that hit the handler. The REST code itself is in the "server" package. The web-site is served through a static content handler. The files themselfs are in htmlFiles. Finally, the web-site has an index page which contains links for the github of the project, short description, this documentation and the "assigment" page. The assigment page uses ajax to the REST service.

Inheritance implementation and strategy

We used inheritance for the RoleSchool class. The inheritance type is joined which is a strategy in which fields that are specific to a subclass are mapped to a separate table than the fields that are common to the parent class, and a join is performed to instantiate the subclass. The classes that inherit from RoleSchool are "Student" ,"Teacher" and "AssistentTeacher". In these classes we do not have an ID or the roleName since they are inherited from the parent class.

This strategy was suitable for us because it defines a table for each class in the inheritance hierarchy to store the local attributes of that class.

How far are we with the project

Besides the basic functionality of the program from points 1-7 we have added all the entities needed for future development so every data structure is present. Because of the time pressure we only managed to add the course functionality so far. Every student, teacher and assistant teacher can be enrolled to a course and a course can be assigned to any of them. In this project we focused on the functionality of the program so in order to ensure that everything works we created at least one test for every method. We treated the exceptions that the methods could throw and we also tested them.
